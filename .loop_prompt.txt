# Looper Rust Port — Autonomous Implementation Loop

## Context

You are implementing a Rust port of **Looper** — an autonomous AI dev team for GitHub repos. The Go codebase is at `legacy/` (166K Go). The Rust target is 16 crates in a Cargo workspace. You have 20 beads (tasks) to complete in order.

**Key principle:** Port FEATURES/BEHAVIOR, not file-for-file. Go code is reference for behavior. Rust implementation uses idiomatic Rust: `enum` for state machines, `thiserror` for errors, `tokio` for async, `serde` for serialization.

## Beads Reference (track progress)

20 beads in `.beads/issues.jsonl`. Check br list / br ready / br blocked to see what's next.

```bash
br list        # All beads with status
br ready       # What's actionable now
br blocked     # What's waiting for dependencies
br dep cycles  # Must be empty
```

## Loop Instructions

1. Check `br ready` to find the next actionable bead
2. Read the bead description (context + spec refs + legacy refs)
3. Read the spec file in `PORTING-TO-RUST/specs/`
4. Read the relevant Go code in `legacy/` for behavior reference
5. Implement the Rust crate in `crates/<crate-name>/`
6. Write tests (unit + integration as per `module-testing-strategy.md`)
7. Run `cargo test` and `cargo clippy` for the crate
8. Mark bead done: `br close <bead-id> --reason "Implemented"`
9. Repeat from step 1

## Testing Requirements

Per `module-testing-strategy.md`:
- Unit tests in every crate (table-driven with rstest)
- Integration tests for cross-crate behavior
- Golden/snapshot tests for CLI/API output (insta)
- Property-based tests for state machines (proptest)
- E2E tests for full system scenarios (fake agent, fake gh)

## Edge Cases to Handle

Per `module-edge-cases.md` (40 edge cases across 8 categories):
- SQLite: BUSY/locked, migration version mismatch, concurrent writes
- System: symlink traversal, disk full, clock skew, zombies, signal differences
- Network: partition, duplicate webhook, node flapping, malformed payload
- Agent: binary not found, OOM crash, hang, large output, empty prompt
- Config: empty file, corrupt file, legacy schema, unknown fields
- Worktree: stale lock, force push failure, protected branch, concurrent creation
- Concurrency: scheduler+webhook+CLI simultaneous writes, dual daemon instances
- Lifecycle: head SHA changes mid-review, auto-merge race, duplicate webhook

## Go vs Rust Behavior Checklist

After each phase, verify:
- [ ] Config: load/save roundtrip matches Go behavior
- [ ] Storage: all 17 migrations run, CRUD matches Go repository methods
- [ ] Service: loop/run/project state transitions match Go
- [ ] GitHub: same gh CLI invocations, same response parsing
- [ ] Agent: same vendor flags, same timeout/parse behavior
- [ ] Scheduler: same tick/claim/discovery flow
- [ ] Runners: same step sequences and state machines
- [ ] API: same endpoint responses, same error codes (19 variants)
- [ ] CLI: same command tree, same output format
- [ ] Daemon: same bootstrap sequence, same recovery pipeline
